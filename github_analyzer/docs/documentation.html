🤖 The 4 Specialized Agents
Repository Structure Agent
Senior Repository Architect
Analyzes file tree, language breakdown, metadata, and top contributors. Maps the tech stack from directory structure.
get_repository get_file_tree get_languages get_contributors
Output → 01_repo_structure.md
Issue Analyzer Agent
Senior DevOps Issue Analyst
Counts open vs closed issues, identifies label patterns (bug/feature/docs), assesses health score with resolution rate.
list_issues label_distribution issue_health_score
Output → 02_issue_analysis.md
PR Analyzer Agent
Senior Code Review & CI/CD Analyst
Measures PR velocity, review depth (avg comments), merge rate, and contribution diversity. Scores review culture.
list_pull_requests avg_comments merge_rate
Output → 03_pr_analysis.md
Branch Analyzer Agent
Senior Git Strategy Analyst
Identifies branching strategy (GitFlow vs GitHub Flow vs trunk), analyzes commit quality, and checks default branch protection.
list_branches get_commits branch_protection
Output → 04_branch_analysis.md

# crew/agents/github_agents.py
from crewai import Agent
from langchain_openai import ChatOpenAI

def create_repo_structure_agent() -> Agent:
    return Agent(
        role="Senior Repository Architect",
        goal="Analyze file tree, languages, and contributors...",
        backstory="You are a veteran software architect with 15+ years...",
        tools=[GitHubRepoStructureTool()],
        llm=ChatOpenAI(model="gpt-4o", temperature=0.2),
        verbose=True,
        allow_delegation=False,
        max_iter=3,
    )
# crew/tools/github_tools.py
from langchain.tools import BaseTool

class GitHubRepoStructureTool(BaseTool):
    name: str = "github_repo_structure"
    description: str = "Analyzes GitHub repository structure..."
    args_schema: Type[BaseModel] = RepoInput

    def _run(self, owner: str, repo: str) -> str:
        results = {}
        results["metadata"] = mcp_client.call_tool(
            "get_repository", {"owner": owner, "repo": repo}
        )
        results["languages"] = mcp_client.call_tool(
            "get_languages", {"owner": owner, "repo": repo}
        )
        return json.dumps(results, indent=2)
# mcp/github_mcp_client.py
class MCPGitHubClient:
    def call_tool(self, tool_name: str, arguments: dict) -> dict:
        if self.use_mcp:
            # Route to actual MCP server
            return self._call_mcp_server(tool_name, arguments)
        else:
            # Fallback: direct GitHub REST API
            return self._call_direct(tool_name, arguments)

    def _call_mcp_server(self, tool_name, arguments):
        response = requests.post(
            f"{self.mcp_url}/tools/call",
            json={"tool": tool_name, "arguments": arguments},
        )
        return response.json()
# crew/crew_assembler.py
from crewai import Crew, Process

class GitHubAnalyzerCrew:
    def run(self, owner: str, repo: str) -> dict:
        agents = create_all_agents()    # 4 agents
        tasks  = create_all_tasks(      # 4 tasks
            agents, owner, repo
        )
        crew = Crew(
            agents=list(agents.values()),
            tasks=tasks,
            process=Process.sequential, # sequential pipeline
            verbose=True,
        )
        crew_output = crew.kickoff()
        return {"success": True, "final_report": ...}
# analyzer/views.py
def run_analysis(request, owner, repo):
    # Called by AJAX from loading.html
    from crew.crew_assembler import GitHubAnalyzerCrew
    crew = GitHubAnalyzerCrew(reports_dir=settings.REPORTS_DIR)
    result = crew.run(owner, repo)

    if result.get("success"):
        return JsonResponse({
            "status": "complete",
            "redirect": f"/report/{owner}/{repo}/",
        })

❓ Interview Q&A
Click any question to expand the answer.
Basic
What is Agentic AI? ▾
Agentic AI refers to AI systems where one or more LLMs act as autonomous agents — taking actions, calling tools, and making decisions to achieve a goal, rather than just generating a single response. The key characteristics are: goal-directedness (works toward a defined objective), tool use (calls external APIs), reasoning loops (thinks through problems step-by-step), and autonomy (decides HOW to achieve the goal). In this project, each of the four agents autonomously decides which tool calls to make, what data to request, and how to synthesize the results into a report.
What is MCP (Model Context Protocol)? ▾
MCP is an open protocol created by Anthropic that standardizes how AI models interact with external tools and data sources. Think of it as "USB for AI tools" — it defines a common interface so that any tool server can expose its capabilities to any compatible AI model. The protocol uses JSON-RPC and defines schemas for tool discovery (list_tools) and execution (call_tool). GitHub has an official MCP server that wraps their REST API. In this project, MCPGitHubClient wraps both the MCP protocol and the direct GitHub REST API as a fallback.
Why use multiple agents instead of one? ▾
Multiple specialized agents outperform a single generalist agent for four reasons: (1) Context window efficiency — each agent only has the context it needs, not the entire analysis; (2) Specialization — an agent with the role "Senior Issue Analyst" produces better issue analysis than a generic agent because the LLM "acts as" that role; (3) Separation of concerns — adding a new analysis dimension means adding a new agent, not modifying a monolith; (4) Sequential context building — in Process.sequential, each agent's output becomes available to subsequent agents, enabling progressively richer analysis.
Intermediate
How does Django integrate with CrewAI? ▾
Django handles HTTP — receiving the repository URL, routing to views, and rendering templates. CrewAI is a Python library, so it's imported directly in the Django view and called synchronously. The run_analysis view is an AJAX endpoint that blocks while CrewAI runs (~60-90 seconds), then returns JSON with a redirect URL. The loading page provides real-time UX feedback while analysis runs via simulated step progress. For production scale, CrewAI execution would move to a Celery task queue with WebSocket updates replacing AJAX polling.
Walk me through what happens when a repo URL is submitted. ▾
1. Browser POSTs to /analyze/ with the URL. 2. analyze view parses "django/django" from the input. 3. View renders loading.html with owner/repo. 4. JS in loading.html immediately POSTs to /run/django/django/. 5. run_analysis view calls GitHubAnalyzerCrew().run("django","django"). 6. Crew creates 4 agents and 4 tasks. 7. Task 1: Agent 1 uses GitHubRepoStructureTool → calls MCP client 4 times → LLM synthesizes → writes 01_repo_structure.md. 8. Tasks 2-4 run sequentially. 9. crew_assembler combines sections → saves django_django_report.md. 10. AJAX returns {"status":"complete","redirect":"/report/django/django/"}. 11. Browser navigates to report. 12. report view reads .md, converts to HTML, renders report.html.
Advanced
How would you make agents run in parallel? ▾
CrewAI supports Process.hierarchical where a manager agent delegates tasks, but true parallelism requires custom implementation. Options: (1) Use asyncio.gather() to run four separate CrewAI instances concurrently; (2) Use concurrent.futures.ThreadPoolExecutor (OpenAI calls are I/O-bound so threads are appropriate); (3) Use LangGraph instead of CrewAI for fine-grained DAG execution control. The trade-off: parallel agents lose sequential context — Task 4 couldn't reference Task 1's findings. For this project, the sequential context sharing is intentional and architecturally valuable.
What are the failure modes and how do you handle them? ▾
(1) OpenAI API failure — wrapped in try/catch, returns error report with guidance; (2) GitHub API rate limit (60 req/hr unauthenticated) — GITHUB_TOKEN raises limit to 5000/hr; (3) Private repositories — token with repo scope required; (4) LLM hallucination — mitigated by low temperature (0.2) and explicit task prompts with real data injected; (5) Long-running analysis — AJAX pattern with 300s timeout; (6) MCP server down — automatic fallback to direct GitHub REST API; (7) Network timeouts — requests.Session with 30s timeout per call.
How does MCP differ from standard LangChain function calling? ▾
Standard LangChain function calling embeds tool schemas directly in the LLM prompt via JSON. MCP is a transport protocol — it separates tool definition (on the MCP server) from tool invocation (by the client). Advantages of MCP: (1) Tools are discovered at runtime via list_tools — no hardcoding; (2) Tools can be shared across different AI systems (Claude, GPT, local models); (3) Tool servers can be updated independently without redeploying the agent; (4) MCP enables tool composition — one MCP server aggregating many upstream APIs. It's the difference between a tightly coupled function call and a loosely coupled microservice.
